<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| task file body → hook annotation check | Untrusted markdown content from agents or humans is read by the hook to check for annotation blocks |
| orchestrator write calls → hook | Every Edit/Write to .planning/work/*.md passes through the hook; the hook is the enforcement point |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | task-state-guard.js annotation check | mitigate | Annotation regex checks for exact format "## QA Results[\s\S]*?Status: FAIL" — an agent or human cannot regress status without a correctly-formatted evidence block present in the file body |
| T-04-02 | Tampering | task-state-guard.js annotation check | accept | An attacker could manually write a fake "## QA Results\n\nStatus: FAIL" block without real QA output, then manually regress status. Risk accepted — this is a developer tooling system, not a security boundary; the annotation is evidence for human reviewers, not a cryptographic proof |
| T-04-03 | Spoofing | receipt signal extraction | accept | A crafted receipt string with uppercase keywords (APPROVED, FAIL) in unexpected positions could mislead the orchestrator. Risk accepted in this phase — orchestrator's signal extraction uses string.includes() which is predictable; this is addressed in Phase 4 Plan 02 implementation |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this plan — zero npm/pip/cargo calls. No slopcheck needed. |
</threat_model>
