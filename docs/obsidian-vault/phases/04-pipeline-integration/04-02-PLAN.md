<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| agent receipt → orchestrator signal extraction | Agent return values contain receipt strings; orchestrator parses signal words from them |
| agent prompt ← orchestrator | Orchestrator constructs prompts from task file content + loop state; task file body could contain adversarial text if a user manually edited it |
| code-reviewer return value → REVIEW-BLOCK extraction | orchestrator extracts text between delimiters from agent output; if delimiters are absent, extraction should fail gracefully |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-04 | Spoofing | Receipt signal extraction in execute.md | mitigate | Signal priority check order (APPROVED → CHANGES_REQUESTED → PASS → FAIL → REJECTED → ERROR) uses plain string includes() checked in priority order; the first match wins. A receipt cannot simultaneously contain APPROVED and FAIL without human tampering — and human tampering is out of scope for this tooling system |
| T-04-05 | Tampering | Agent prompt injection via task file body | accept | Task file body content is included in agent prompts (stage context, prior receipts). A manually-crafted task file body could attempt to redirect agent behavior. Risk accepted — agents operate under strict system prompts and repo isolation; prompt injection into markdown orchestrator context is a residual risk for a dev tooling system, not a production security boundary |
| T-04-06 | Elevation of Privilege | code-reviewer disallowedTools bypass | accept | code-reviewer has disallowedTools: Write, Edit, Bash enforced by Claude Code runtime. The orchestrator cannot bypass this. If Claude Code runtime enforcement is bypassed, all bets are off — this is not a planner-addressable threat |
| T-04-07 | Denial of Service | Unbounded loop in CHANGES_REQUESTED path | mitigate | cr_cycle cap = 2 with Retry/Skip/Abort gate prevents indefinite CHANGES_REQUESTED cycling. QA cap = 3 and TLC cap = 2 also prevent infinite loops |
| T-04-08 | Tampering | git diff stat in code-reviewer prompt | accept | git diff --stat output could contain adversarial content from commit messages. Risk accepted — the stat output is file names and line counts only; commit message text does NOT appear in --stat output. Low risk. |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this plan — zero npm/pip/cargo calls. No slopcheck needed. |
</threat_model>
